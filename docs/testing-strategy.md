# テスト戦略と自動テスト環境構築

## 1. 背景と目的

本ドキュメントは、NestJS + React（Vite）のモノレポ環境において、テスト戦略の策定から CI/CD パイプラインでの自動テスト環境構築まで、実践した記録である。

### なぜテスト自動化に取り組んだか

- 手動テストでは回帰テストのコストが開発規模に比例して増大する
- コードの変更が既存機能を壊していないことを、毎回人手で確認するのは現実的でない
- テストを自動化し CI に組み込むことで、PR 単位で品質を担保できる体制を構築した

### 対象プロジェクト構成

```
skill-up-testing-git/          # pnpm workspace モノレポ
├── packages/backend/          # NestJS（TypeScript）
├── packages/frontend/         # React + Vite（TypeScript）
└── e2e/                       # Playwright（API テスト + E2E テスト）
```

| レイヤー | テストツール | 用途 |
|----------|-------------|------|
| バックエンド | Vitest | ユニットテスト |
| フロントエンド | Vitest + Testing Library | コンポーネントテスト |
| API 統合 | Playwright（request） | HTTP 経由の API テスト |
| E2E | Playwright（page） | ブラウザ操作の E2E テスト |

## 2. テスト戦略

### テストピラミッドの考え方

テストは「ピラミッド型」で設計する。下層ほど数が多く、高速で、安定している。

```
        /  E2E  \          少数・遅い・壊れやすい
       /──────────\
      / 統合テスト  \       中程度
     /──────────────\
    / ユニットテスト   \    多数・高速・安定
   /────────────────────\
```

| 層 | 目的 | 速度 | 安定性 | 本プロジェクトでの実装 |
|----|------|------|--------|----------------------|
| ユニットテスト | 関数・クラス単体のロジック検証 | 非常に速い（ms） | 高い | Vitest |
| 統合テスト | 複数コンポーネントの結合動作確認 | やや遅い（秒） | 中程度 | Playwright API テスト |
| E2E テスト | ユーザー操作の再現 | 遅い（秒〜分） | 壊れやすい | Playwright ブラウザテスト |

### 採用方針

- **ユニットテストを最も厚く書く。** ビジネスロジック（計算、バリデーション、条件分岐）は全てユニットテストでカバーする。
- **統合テストは API の結合確認に絞る。** ルーティング、リクエスト/レスポンス形式、エンドポイント間の連携を確認する。
- **E2E テストは主要導線のみ。** 全画面を網羅するのではなく、ユーザーの主要フローに限定する。

---

### ユニットテスト（Vitest）の設計指針

Vitest を採用した理由は、Vite ベースのプロジェクトとの親和性が高く、設定がシンプルな点にある。NestJS ではデコレータを使用するため、`unplugin-swc` で SWC 変換を行っている。

#### テストの書き方の原則

**1. テストごとに状態を初期化する**

```ts
let service: CartService;

beforeEach(() => {
  service = new CartService();
});
```

`beforeEach` でインスタンスを再生成し、テスト間の状態汚染を防ぐ。あるテストで追加したデータが別のテストに影響する問題を根本から排除する。

**2. テストデータはヘルパー関数で生成する**

```ts
const createItem = (overrides?: Partial<CartItem>): CartItem => ({
  id: "item-1",
  name: "りんご",
  price: 100,
  quantity: 1,
  ...overrides,
});
```

テストごとに必要な部分だけ上書きすることで、テストの意図が明確になる。

**3. テストケースの観点**

| 観点 | 例 |
|------|-----|
| 正常系 | 期待通りの入力で期待通りの結果が返る |
| 境界値 | 0、1、上限値 |
| 異常系 | 存在しないID、不正な値でエラーがスローされる |
| 副作用 | 状態が正しく変化しているか（防御的コピーの確認等） |

**4. `describe` でメソッド単位にグループ化し、正常系 → 異常系の順で書く**

```ts
describe("CartService", () => {
  describe("addItem", () => {
    it("新規アイテムを追加できる", ...);
    it("同じIDのアイテムは quantity が加算される", ...);
    it("異なるIDのアイテムはそれぞれ追加される", ...);
  });

  describe("removeItem", () => {
    it("存在するアイテムを削除できる", ...);
    it("存在しないIDで Error をスローする", ...);
  });
});
```

### 統合テスト（Playwright API テスト）の設計指針

Playwright の `request` コンテキストを使い、ブラウザを起動せずに HTTP リクエストでバックエンド API をテストする。

#### ユニットテストとの使い分け

| 観点 | Vitest ユニットテスト | Playwright API テスト |
|------|----------------------|----------------------|
| 対象 | 関数・クラス単体 | HTTP API 全体 |
| サーバー起動 | 不要 | 必要 |
| 速度 | 非常に速い | やや遅い |
| 検証できること | ロジックの正しさ | ルーティング、シリアライズ、HTTP ステータス |
| 発見できるバグ | 計算ミス、条件分岐漏れ | エンドポイント設定ミス、リクエスト/レスポンス形式の不整合 |

#### 実装のポイント

- `playwright.config.ts` で `projects` を分離し、API テストと E2E テストを独立して実行できるようにする
- `test.beforeEach` でテストごとにカートをクリアし、テスト間の依存を排除する
- 単一操作の確認だけでなく、複数操作を組み合わせたフローテストも書く

```ts
// playwright.config.ts（抜粋）
projects: [
  {
    name: "api",
    testMatch: "**/*.api.spec.ts",
    use: { baseURL: "http://localhost:3000" },
  },
  {
    name: "e2e",
    testMatch: "**/*.e2e.spec.ts",
    use: { baseURL: "http://localhost:5173", browserName: "chromium" },
  },
],
```

---

### E2E テスト（Playwright ブラウザテスト）の設計指針

`page` コンテキストを使い、実際のブラウザでフロントエンドの動作を確認する。

- 主要なユーザーフロー（ページ表示、主要操作）に限定する
- ロジックの検証はユニットテストに任せ、E2E では「画面が正しく表示されるか」「操作が通るか」に集中する
- テストが壊れやすくならないよう、DOM 構造への過度な依存を避ける

## 3. TDD の実践（t-wada 方式）

### TDD とは

テスト駆動開発（Test-Driven Development）は、実装の前にテストを書く開発手法。

```
通常の開発：  実装 → テスト → 動作確認
TDD：        テスト → 実装 → リファクタ → テスト → 実装 → ...
```

### Red → Green → Refactor サイクル

```
🔴 Red      失敗するテストを「1つだけ」書く（まだ実装しない）
🟢 Green    テストが通る「最小限の」コードを書く（きれいさは無視）
🔵 Refactor テストが通ったまま、コードをきれいにする
```

### 実践記録：CartService の割引機能

CartService に割引機能（%割引、固定額割引、最低購入額条件、下限0制御）を TDD で追加した。以下はサイクルの記録。

#### サイクル 1：割引率の基本

**🔴 Red** → `applyDiscount` メソッドがまだ存在しない状態でテストを書く。

```ts
it("10%割引を適用できる", () => {
  service.addItem(createItem({ price: 1000, quantity: 1 }));
  service.applyDiscount({ type: "percent", value: 10 });
  expect(service.getTotal()).toBe(900);
});
```

テスト実行 → `applyDiscount is not a function` で失敗。

**🟢 Green** → `applyDiscount` メソッドと `getTotal` への割引反映を最小限で実装。

**🔵 Refactor** → この段階では不要。次のサイクルへ。

#### サイクル 2〜5

| サイクル | 機能 | テスト内容 |
|---------|------|-----------|
| 2 | 固定額割引 | 500円割引の適用 |
| 3 | 最低購入額条件 | 条件未達で割引なし / 条件達成で割引適用 |
| 4 | 割引後の下限 | 固定額割引で合計が0未満にならない |
| 5 | 割引のクリア | `clear()` で割引もリセットされる |

各サイクルで必ず 🔴 Red（失敗）を確認してから 🟢 Green に進んだ。

### TDD の価値

テストを先に書くことで以下の効果を実感した。

- **使う側の視点で API を設計できる。** テストを書く時点で「このメソッドはどう呼ばれるべきか」を考えるため、使いやすいインターフェースになる
- **必要最小限の実装に集中できる。** テストを通すためだけのコードを書くので、YAGNI（You Ain't Gonna Need It）が自然に守られる
- **リファクタリングを安心して行える。** テストが通る状態を維持しながらコードを整理できるため、「壊してしまうかもしれない」という恐怖がなくなる

### TDD のアンチパターン

| ❌ やってはいけないこと | 理由 |
|------------------------|------|
| テストを書く前に実装する | TDD のリズムが崩れ、テストが形骸化する |
| 一度に複数のテストを書く | 1つ失敗 → 1つ通すのサイクルが回らなくなる |
| Green で完璧なコードを書こうとする | 最小限でOK。きれいにするのは Refactor フェーズ |
| Red を確認せずに進む | テストが実際に機能を検証しているかの確認を飛ばすことになる |

## 4. CI/CD パイプラインでのテスト自動化

### GitHub Actions ワークフロー構成

PR を作成・更新したときに、以下が自動実行される。

```
PR 作成 / 更新
  │
  ├─ Checkout
  ├─ pnpm install（キャッシュ済みなら高速）
  │
  ├─ pnpm coverage
  │   ├─ backend: vitest run --coverage
  │   │   └─ 閾値チェック（80% 未満 → CI 失敗）
  │   └─ frontend: vitest run --coverage
  │       └─ 閾値チェック
  │
  ├─ カバレッジレポート → PR サマリに表示
  │
  ├─ Playwright API テスト
  │
  └─ （失敗時のみ）Playwright レポートをアーティファクトに保存
```

### 主要な設計判断

| 判断 | 理由 |
|------|------|
| `--frozen-lockfile` | lockfile と package.json の不一致を CI で検出する |
| `if: always()` でカバレッジレポート | テスト失敗時でもカバレッジレポートは表示する |
| CI では `--project=api` のみ | E2E はフロントエンド起動が必要で遅い。API テストだけに絞る |
| 失敗時のみアーティファクト保存 | 成功時は不要。ストレージ節約 |

### カバレッジ閾値の設定

Vitest の設定でカバレッジ閾値を設定し、下回った場合に CI が失敗する。

```ts
// vitest.config.ts（抜粋）
coverage: {
  provider: "v8",
  reporter: ["text", "json-summary", "json", "html"],
  thresholds: {
    statements: 80,
    branches: 80,
    functions: 80,
    lines: 80,
  },
},
```

- `json-summary` : カバレッジレポートアクション用
- `json` : 詳細なファイル別レポート用
- `html` : ローカルでの確認用
- `thresholds` : 下回ると `vitest run --coverage` が exit code 1 を返す

### カバレッジレポートの PR 表示

`davelosert/vitest-coverage-report-action` を使用し、PR の Checks タブにカバレッジサマリを表示する。これにより、ファイル単位のカバレッジを確認でき、テストが不足している箇所を素早く特定できる。

## 5. 成果と今後の改善案

### 成果

| 指標 | 結果 |
|------|------|
| ユニットテスト数 | 29件（AppService 7件 + CartService 22件） |
| Playwright API テスト数 | 7件 |
| Playwright E2E テスト数 | 2件 |
| バックエンドカバレッジ | cart.service.ts 100%（Stmts/Branch/Funcs/Lines） |
| CI 実行時間 | 約 47〜49 秒（キャッシュ適用後） |

### 今後の改善案

- **フロントエンドのテスト強化。** 現状は Vite デフォルトページのみ。カート UI を実装し、コンポーネントテストと E2E テストを追加する
- **テストデータのファクトリパターン導入。** テストデータ生成をより体系的に管理する（例：`fishery` や `faker` の導入）
- **Playwright のビジュアルリグレッションテスト。** スクリーンショット比較によるUIの回帰テストを検討する
- **テストの並列実行最適化。** テスト数が増えた場合、CI の並列実行やシャーディングを検討する
